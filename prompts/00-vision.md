# Sketchapedia — Full Vision Reference

This document is a long-form reference. Each task prompt embeds a short version; read this file when you need more context.

## The core idea: Model-as-a-Renderer (MaaR)

For 30 years the web has been a document model painted by a browser. HTML declares structure, CSS paints it, JavaScript mutates it. The rendering engine is deterministic and rectangular.

Sketchapedia proposes a different renderer: a **generative AI pipeline that produces UI as imagery**. Interfaces can be hand-drawn maps that zoom into cross-sections of landmarks, scientific diagrams that morph into cinematic footage, isometric cityscapes that scrub through historical eras, comic-book dashboards that redraw themselves to explain a database migration. None of this is HTML. It is pixels produced by models, keyed to user intent.

## The inspiration (six reference videos)

1. **Essential Guide to Paris** — stylized hand-drawn map zooms into an Eiffel Tower cross-section and then into a Le Jules Verne reservation interface with interactive booking timeline.
2. **Why Does Ice Float** — scientific diagram of liquid vs. solid water at molecular level; on interaction morphs into photorealistic forest/water footage.
3. **Hydrogen Bond Geometry** — continues the diagram; user clicks a text box and the entire page re-lays itself to explain "Expansion Upon Freezing" with new charts.
4. **Evolution of Times Square** — interactive architectural timeline; dragging a scrubber updates the isometric cityscape through different eras.
5. **Project Dashboard** — comic-book-style software-engineering dashboard with a "To-Do List Application" schema. Clicking "Denormalize Schema" draws a Before/After diagram.
6. **Codebase Architecture** — companion to #5, same interaction model.

## The problem with naïve MaaR

Streaming live 24fps video generated by a heavy model for every micro-interaction costs dollars per minute per user. Tail latency is 100–500ms. Video models hallucinate text frame-to-frame. Pixel streams are unreadable to screen readers, password managers, IMEs, translate, find-in-page, copy-paste. Demos from early pioneers were sped up and lost fidelity on text.

## Sketchapedia's answer: hybrid generative-logical

Three layers, back to front on the client:

1. **Canvas layer** — a generated keyframe image (or short looping video) drawn to `<canvas>`. The keyframe is content-addressed and cached at CDN edge. Most interactions never hit a GPU.
2. **Effects layer** — WebGL/WebGPU cursor reactivity, hover highlights, particle feedback. Runs locally; no network.
3. **DOM overlay layer** — invisible real `<input>`, `<button>`, ARIA-labeled elements positioned from a hitmap JSON. This is what the user actually interacts with. Text input, IME composition, password managers, screen readers, keyboard nav, focus rings — all solved by using the platform instead of simulating it.

Backend pipeline for a *novel* intent (cache miss):

1. LLM receives current scene id, intent, state delta, reference image hash. Emits: next scene layout spec + hitmap JSON + image prompt. Structured output, single call.
2. Image model renders next keyframe conditioned on current keyframe (IP-Adapter / ControlNet reference) to preserve style.
3. Video model renders a 0.5–1.5s transition clip from current → next keyframe.
4. Vision model grounds the hitmap against the rendered image to correct coordinates (the alignment problem — the LLM can say a button is at (298, 475) but the image model draws it at (312, 480); the vision pass closes that gap).
5. Artifacts stored content-addressed in object store; CDN-pushed; cache key announced to client.

Cache-first steady state: most user paths hit CDN, not GPUs.

## The trickiest problem

Not text input (solved: DOM overlay cheat). Not temporal consistency (solved by not streaming frames — we render keyframes + short transitions). The real hard problem is **hitmap-to-pixel alignment**. If the hitmap says "click here" but the pixels show the button somewhere else, UX collapses. Mitigations:

- **Layout-first** — ControlNet regional prompting forces the image model to respect LLM-declared bounding boxes. Robust but limits aesthetic freedom, which defeats the point.
- **Vision post-pass** — generate the image freely, then use Florence-2 / Grounding DINO to recover accurate bounding boxes from pixels. Doubles inference cost; reliable.
- **Generous hit regions + snap** — cheap fallback; degrades when regions are dense.

Sketchapedia combines: LLM emits a hitmap *prior*, image is generated with light regional guidance, vision post-pass corrects. The final hitmap is what the client sees.

## Non-goals

- Replacing HTML universally. Sketchapedia is for expressive generative UIs, not bank statements.
- Frame-level streaming from heavy models. Not affordable, not consistent.
- Pixel-perfect reproducibility. Slight stylistic drift between users on cache misses is acceptable; layout + hitmap drift is not.
- Avoiding the DOM. The DOM overlay is a feature, not a compromise — it's how accessibility and input fidelity work.

## Glossary

- **Scene**: one unit of UI; `{ keyframe_image, hitmap, state_schema, transition_in? }`.
- **Hitmap**: JSON list of `{ id, polygon_or_bbox, role, aria_label, intent, input_kind? }` items.
- **Intent**: a named action triggered by an interactive region (e.g. `open_reservation`, `scrub_to_year(1928)`).
- **Transition**: short video clip (or generated frame sequence) between two scene keyframes.
- **Content-addressed key**: `sha256(canonical_json({ prev_scene_id, intent, state_delta, style_ref }))` — identical inputs produce identical output artifacts, so caching works across users.
- **MaaR**: Model-as-a-Renderer.

## Quality bar (non-negotiable)

- No mocks. No stubs. No TODOs left behind. No "simplified for demo" paths.
- Every module has real tests that exercise real behavior.
- Every external integration (models, object store, Redis, CDN) talks to a real instance in CI.
- Strict TypeScript. Proper error types. Structured logs. Telemetry on every boundary.
- Accessibility is a first-class requirement in every client-facing primitive.
- Security is a first-class requirement in every server-facing primitive.

If a prompt's acceptance criteria can be met by a mock or a stub, the prompt has failed — treat "no mocks" as a test of the implementation, not a preference.
