# 26 — Security & Sandboxing

## Project context

Sketchapedia executes user-influenced data through an LLM, an image model, and a video model, and sends imagery back to the user's browser. This creates three attack surfaces: **prompt injection** manipulating the LLM, **output-based attacks** (malformed hitmap coordinates, XSS via ARIA labels, tainted images), and **infrastructure abuse** (cost exhaustion, scraping cached artifacts). See `prompts/00-vision.md`.

## Your task

Implement `packages/security/` plus a top-level `SECURITY.md` specifying the threat model, mitigations, and responsible-disclosure process. The package provides runtime defenses imported by the gateway, orchestrator, and models; the SECURITY.md documents policy.

## Threat model (must be fully enumerated in SECURITY.md)

1. **Prompt injection**: user intents, state deltas, or tenant style guides contain instructions that attempt to override system prompts, exfiltrate system-prompt contents, produce policy-violating imagery, or hijack the LLM's tool output.
2. **Hitmap tampering**: the LLM or a compromised intermediate service emits hitmap coordinates that point users at malicious outgoing links, or that overlay invisible `<a href="attacker.com">` regions over innocuous-looking buttons.
3. **XSS via ARIA**: aria-labels generated by the LLM contain HTML or script; injected into the DOM overlay as `textContent` is safe, but `innerHTML` paths must be audited.
4. **Cost exhaustion**: attacker fires rapid unique intents to force cache misses.
5. **Scraping / artifact theft**: attacker harvests signed URLs to mirror tenant content.
6. **Dependency supply chain**: compromise of a model weight, a LoRA, or an npm package.
7. **Session hijacking**: stolen auth tokens, replayed session IDs.
8. **Denial of service**: flood gateway, flood orchestrator, flood models, exhaust quotas.

## Mitigations (implement in `packages/security/`)

- **Prompt isolation wrapper**: `wrapUserInput(input)` returns a quoted block with delimiter randomization and an explicit reminder of system intent; used by prompt 16's LLM service.
- **Tool-output validator**: every LLM tool output (LayoutPlan) is Zod-validated + additional content-policy checks: `intent.href` must be a same-origin route or an allowlisted host; `ariaLabel` stripped of HTML tags; coordinates clamped to keyframe bounds.
- **DOM safe insertion**: export a `textNode(str)` helper used by the overlay to insert text; forbid `innerHTML` via an ESLint rule.
- **Intent rate limit** (gateway): token bucket per user/tenant; stricter bucket for cache-miss intents (detected when `(prev, intent, state)` is novel).
- **Signed URLs** (prompts 20, 21): short TTL + tenant-scoped; rotation weekly.
- **Content policy**: central `ContentPolicy` class with `shouldBlock(kind, text)` returning typed reasons; wired into LLM pre- and post-call, image-model prompt validator.
- **Image safety checker**: runtime NSFW filter in prompt 17; this package coordinates policy decisions.
- **Auth**: JWT validation with rotation; refresh-token flow; CSRF for any HTTP endpoint that isn't a WS; strict CORS.
- **Secrets hygiene**: `checkEnv()` that boots each service, validates required env, and verifies no secret is printed in logs. Redact known secret patterns in log sinks.
- **Dependency scanning**: `pnpm audit` + `osv-scanner` in CI; block PRs with critical CVEs.
- **SBOM**: generated via `@cyclonedx/cyclonedx-npm` per package; published in releases.
- **Pinned model weights**: each model server records a hash of the weights on boot; mismatch aborts start.
- **Denylist**: a tenant-level denylist of intents and style hints; enforced in the orchestrator.

## Public API

```ts
import { wrapUserInput, ContentPolicy, sanitizeLayoutPlan, textNode, validateHref, ensureEnv, SecurityError } from "@sketchapedia/security";
```

## Implementation mandates

- Stateless where possible; rules loaded from policy files under `packages/security/policies/*.yaml`.
- Zero regex-based HTML parsing — use `linkedom` or `parse5` when DOM parsing is required, or pure text-node insertion.
- Logs never include raw user input; hash + length instead.
- All `SecurityError` paths surface structured reasons for observability.
- Fuzz tests (`@fast-check/vitest`) for every validator.

## Test plan

- Red-team suite: 100 adversarial prompts attempting injection, role override, data exfil; all neutralized.
- DOM fuzz: 10 000 random strings passed through overlay; assert no script executes (use CSP report-only + Playwright assertion).
- Signed URL brute force: 1M random URLs; all rejected.
- CSP report endpoint collects violations in CI runs of reference apps; zero violations expected.
- Dependency scan green on main branch.

## Deliverables

- `packages/security/src/*`.
- `packages/security/policies/*.yaml`.
- Top-level `SECURITY.md` — threat model + disclosure + contact.
- `.github/workflows/security.yml` — `pnpm audit` + `osv-scanner` + container image scanning.
- CSP header defaults for reference apps.

## Acceptance criteria

- Red-team suite 100% neutralized.
- Dependency scan green.
- SBOM generated for every package on release.
- CSP on reference apps reports zero violations in E2E runs.

## Non-goals

- No cryptography primitives (use `@noble/*` and WebCrypto).
- No runtime sandboxing of third-party JS (out of scope for v1).
- No bug-bounty operational tooling beyond the `SECURITY.md` contact.
